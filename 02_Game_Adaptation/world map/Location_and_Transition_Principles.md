# Location and Transition Principles (Принципы мест и переходов)

## Кратко

Этот документ фиксирует рабочее понимание мест, переходов и маршрутов для `MadeInAbyss_RPG`.

Документ нужен как внутренняя опора перед созданием карточек мест и переходов.

## Базовая модель

Место — открытая сцена исследования и выбора.

Переход — направленная сцена перемещения.

Маршрут — цепочка мест и переходов.

В Obsidian мастер видит всю структуру через ссылки и граф. Игроки рисуют свою карту сами и могут ошибаться.

## Место

Место — область, где персонажи могут относительно свободно перемещаться по горизонтали.

Место может быть маленьким уступом, площадкой, стоянкой, влажной трещиной или большим пространством вроде цветочного поля.

Главный признак места:

```
Внутри места перемещение не является главным испытанием.
```

В месте персонажи могут осмотреться, сделать привал, собрать образцы, найти следы, выбрать переход, вернуться позже или изменить состояние зоны своими действиями.

Если место становится слишком большим, насыщенным или запутанным, его лучше разделить на несколько меньших мест и соединить переходами.


## Переход

Переход — направленная сцена перемещения между местами или между другими переходами.

Переход не обязательно является тоннелем. Это может быть спуск, подъём, карниз, склон, корневой путь, трещина, проход через заросли, мост, лаз или обход.

Главная формула:

Переход — это туннель из точек.

Это значит, что персонажи двигаются через ограниченное пространство и последовательно проходят несколько точек прохождения.

Главный признак перехода:

```
Форма пути ограничивает свободу движения.
```

Переход может иметь цену, находки, скрытые ответвления, последствия и изменяемое состояние.

Переход не должен ограничивать игроков списком готовых способов прохождения. Он описывает среду, точки прохождения, условия и последствия.

## Связи

Места и переходы соединяются как граф.

Базовая связь:

```
Место → Переход → Место
```

Допустимы более сложные связи:

```
Место → Переход → Переход → МестоМесто → Переход → Скрытый переход → Скрытое место
```

Из места в другое место нельзя попасть напрямую без перехода.

Если в место можно попасть через переход, обычно через него же можно попытаться выбраться обратно, но условия обратного пути могут измениться.

## Скрытые переходы

Скрытые переходы можно ссылать напрямую.

Obsidian используется только мастером, поэтому нет необходимости прятать ссылки от графа.

Пример:

```
Скрытый переход: [[T014_Narrow_Root_Crack]].
```

Скрытость означает не “ссылка невидима”, а “игроки не знают об этом переходе, пока не обнаружат его в игре”.

## Состояние переходов

Для состояния переходов используем чекбоксы.

Они нужны только там, где переход реально может измениться.

Пример:

```
## Состояние перехода- [ ] Верхний зацеп повреждён.- [ ] Верёвка оборвана.- [ ] Обратный путь стал опаснее.- [ ] Переход привлёк внимание фауны.
```

Чекбоксов не должно быть слишком много. Обычно достаточно 2–4.

## Состояние мест

У мест тоже может быть состояние, но использовать его нужно только при необходимости.

Пример:

```
## Состояние места- [ ] Флора повреждена.- [ ] Здесь оставлен заметный запах.- [ ] Фауна уже привлечена.- [ ] Место больше не подходит для привала.
```

## Цена и варианты

Цена может быть и у места, и у перехода.

Место может иметь цену за взаимодействие:

```
сорвали опасное растение;нашумели;оставили запах;потревожили фауну;повредили опору;испортили образец.
```

Переход может иметь цену за способ прохождения:

```
использовали верёвку;прыгнули;пошли с большим предметом;прошли шумно;пошли без страховки;повредили обратный путь.
```

Главное правило:

```
Цена возникает не просто за факт перехода, а за выбранный способ действия.
```

## Карта и граф

Общей нарисованной карты у мастера может не быть.

Основная мастерская карта — это Obsidian-граф из карточек мест и переходов.

Игроки рисуют собственную карту самостоятельно. Их карта может быть неполной, неточной или ошибочной.

Для мастера структура должна оставаться строгой:

```
каждое место имеет ссылки на переходы;каждый переход имеет ссылки на связанные места;связи видны через граф Obsidian.
```

## Папки

Карточки мест и переходов хранятся по слоям.

Пример:

```
02_Game_Adaptation/Layer_1/Locations/
02_Game_Adaptation/Layer_1/Transitions/
02_Game_Adaptation/Layer_2/Locations/
02_Game_Adaptation/Layer_2/Transitions/
02_Game_Adaptation/Layer_3/Locations/
02_Game_Adaptation/Layer_3/Transitions/
```

Так структура остаётся понятной и не смешивает разные слои.

## Наименование мест

Места получают короткое уникальное имя с ID.

Формат:

```
L001_Old_Training_Ledge
L002_Wet_Crack
L003_Amberberry_Shelf
```

`L` означает location.

Число даёт уникальность.

Название остаётся художественным, но понятным.

## Наименование переходов

Переходы получают короткое уникальное имя с ID.

Формат:

```
T001_Old_Rope_Drop
T002_Mossy_Crack_Passage
T003_Outer_Stone_Traverse
```

`T` означает transition.

Число даёт уникальность.

Название описывает характер пути, а не обязательно содержит точки `from` и `to`.

Связь откуда и куда указывается в YAML и тексте карточки.

Пример:

```
from:  - L001_Old_Training_Ledgeto:  - L002_Wet_Crack
```

И в теле:

```
Соединяет [[L001_Old_Training_Ledge]] и [[L002_Wet_Crack]].
```

## Почему не использовать длинные имена переходов

Не использовать формат:

```
Old_Training_Ledge_to_Wet_Crack_Slippery_Root_Descent
```

Причины:

```
слишком длинно;плохо читается;путает граф;неудобно на телефоне;переходов будет много.
```

Лучше короткое имя + связи внутри карточки.

## Индекс слоя

После создания набора мест и переходов для слоя можно сделать отдельный индекс.

Пример:

```
02_Game_Adaptation/Layer_1/Layer_1_Route_Index.md
```

Индекс нужен как оглавление слоя. Он не обязан подробно описывать все связи, потому что связи видны через карточки и граф.

## Рабочий принцип

```
Не перегружать карточки.Описание для игроков — атмосферное и понятное.Информация для мастера — короткая, однозначная и практичная.Если зона становится слишком сложной, разделить её на несколько мест и переходов.
```
